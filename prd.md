# Product Requirements Document (PRD)
## Помощник по подаче виз в приложении Т-Путешествия

**Версия:** 1.0  
**Дата:** 28 февраля 2025

---

## 1. Executive Summary

### Problem Statement
Пользователи при подаче виз в США, Шенген, Великобританию и Японию многократно вводят одни и те же данные в разные формы, допускают ошибки и опечатки, что снижает шансы одобрения. Данные не сохраняются между подачами и странами.

### Proposed Solution
Встроенный в приложение Т-Путешествия помощник на базе LLM: пользователь один раз загружает документы и вводит персональные данные; система автоматически заполняет все поля заявок для выбранных юрисдикций, валидирует данные и сохраняет профиль для последующих подач.

### Success Criteria
- **Сокращение времени на заполнение заявки:** не менее 60% по сравнению с ручным вводом (замер на тестовой выборке форм).
- **Доля корректно заполненных полей:** ≥ 98% при проверке по чек-листам официальных требований по странам.
- **Повторное использование профиля:** не менее 40% пользователей используют сохранённый профиль при второй и последующих подачах (за 6 месяцев после запуска).
- **Конверсия в подачу:** рост доли пользователей, завершивших подачу после начала заполнения, на ≥ 25% относительно текущего воронки (если метрики есть) или целевой benchmark.
- **NPS/удовлетворённость:** средняя оценка помощника по визам ≥ 4.0 из 5 в in-app опросе.

## 2. User Experience & Functionality

### 2.1 User Personas

| Персона | Описание | Ключевые потребности |
|--------|----------|----------------------|
| **Частый путешественник** | Подаёт визы 1–2 раза в год в разные страны | Единый профиль, минимум повторного ввода |
| **Первичный заявитель** | Впервые оформляет визу | Пошаговая подсказка, проверка полноты документов |
| **Семья/группа** | Подаёт заявки на нескольких человек | Копирование общих данных, управление несколькими профилями |
| **Деловой путешественник** | Регулярные поездки по работе | Скорость, сохранение данных для следующих поездок |

### 2.2 User Stories

**US-1.** Как пользователь приложения Т-Путешествия, я хочу один раз ввести свои персональные данные и загрузить документы, чтобы система подставила их во все поля заявки на визу и я не дублировал ввод.

- **Acceptance Criteria:**
  - Есть экран/раздел «Мой визовый профиль» с полями: ФИО, дата рождения, гражданство, паспорт, адрес, работа/учеба, финансы, поездки (по объёму требований каждой страны).
  - Поддерживается загрузка документов: паспорт, фото, справки (форматы: PDF, JPG, PNG; макс. размер файла — TBD, рекомендовано до 10 MB).
  - При выборе страны и типа визы LLM заполняет все поля формы заявки на основе профиля и извлечённых из документов данных.
  - Пользователь видит, какие поля заполнены автоматически и может их отредактировать перед отправкой/экспортом.

**US-2.** Как пользователь, я хочу, чтобы система проверяла полноту и соответствие данных требованиям выбранной страны, чтобы снизить риск отказа из-за ошибок или недостающих данных.

- **Acceptance Criteria:**
  - Перед «отправкой»/экспортом заявки показывается чек-лист: обязательные поля, формат дат, срок действия паспорта, размер фото и т.д.
  - Отображаются предупреждения и рекомендации по исправлению (текст и, при необходимости, ссылки на официальные требования).
  - Критичные ошибки блокируют переход к следующему шагу до исправления.

**US-3.** Как пользователь, я хочу, чтобы мои визовые данные сохранялись в приложении для последующих подач, чтобы не вводить их заново и быстрее подавать в другую страну или продлевать визу.

- **Acceptance Criteria:**
  - Профиль сохраняется на устройстве и/или в облаке (с учётом согласия и политики конфиденциальности).
  - При новой подаче пользователь может выбрать «использовать сохранённый профиль» и при необходимости обновить только изменившиеся данные.
  - Реализована синхронизация между устройствами при наличии аккаунта (TBD: объём MVP).

**US-4.** Как пользователь, я хочу выбрать страну назначения (США, Шенген, Великобритания, Япония) и тип визы, чтобы получить форму и подсказки, релевантные именно этой стране.

- **Acceptance Criteria:**
  - В интерфейсе доступен выбор: США, Шенген, Великобритания, Япония.
  - Для каждой страны отображаются поддерживаемые типы виз (туристическая, деловая и т.д. — список уточняется по официальным источникам).
  - Структура полей и подсказки соответствуют актуальным требованиям (источники и процесс обновления — в технической спецификации).

### 2.3 Non-Goals (в рамках данной версии)

- Оплата визовых сборов и запись в консульство/ВЦ через приложение (только подготовка данных и форм).
- Юридическая консультация и гарантия одобрения визы.
- Поддержка стран помимо США, Шенген, Великобритания, Япония в MVP.
- Полная автоматическая подача в официальные порталы (если это потребует интеграции с государственными системами) — в MVP фокус на заполнении и экспорте данных/документов для последующей ручной подачи пользователем.
- Распознавание рукописных документов в MVP (приоритет — машиночитаемые PDF/фото).

## 3. AI System Requirements

### 3.1 Tool Requirements

- **LLM API:** модель с поддержкой структурированного вывода (JSON/поля форм), достаточным контекстом для инструкций и примерами (TBD: конкретный провайдер и модель; рекомендуется версия с function calling / structured output).
- **Document understanding:** извлечение текста и структурированных данных из PDF и изображений (паспорт, справки) — OCR/vision API или специализированный сервис; при необходимости — предобученный NER для имён, дат, номеров документов.
- **Хранение промптов и шаблонов:** версионируемое хранилище системных промптов, правил маппинга «профиль → поля формы» по странам и типам виз.
- **Валидация:** правила и форматы по странам (регулярные выражения, справочники, лимиты) — конфигурируемые и обновляемые без деплоя кода (где возможно).

### 3.2 Evaluation Strategy

- **Точность заполнения полей:** выборка из N заявок (N ≥ 50 на страну в тесте); сравнение с эталоном, заполненным экспертом по официальным требованиям; метрика — доля полей без ошибок (целевой порог ≥ 98%).
- **Извлечение из документов:** точность/полнота извлечения ключевых полей (ФИО, даты, номера) из тестового набора сканов паспортов и справок (целевой порог — TBD по результатам пилота).
- **Регрессия при смене версии LLM/промптов:** автоматические тесты на фиксированных примерах профилей и документов; алерт при падении доли корректно заполненных полей ниже порога.
- **Пользовательская обратная связь:** кнопки «Всё верно» / «Есть ошибка» по блоку полей; сбор анонимизированной статистики для дообучения и улучшения промптов.

## 4. Technical Specifications

### 4.1 Architecture Overview

```
[Приложение Т-Путешествия]
         │
         ▼
[Визовый помощник — UI]
         │
         ├──► [Визовый профиль] ◄──► (Локальное/облачное хранилище)
         │
         ├──► [Загрузка документов] ──► [Document parsing / OCR]
         │                                      │
         │                                      ▼
         ├──► [Выбор страны/типа визы] ──► [Маппинг полей + шаблоны]
         │                                      │
         │                                      ▼
         └──► [LLM: заполнение формы] ◄── (Профиль + извлечённые данные + правила страны)
                    │
                    ▼
              [Валидация] ──► [Предпросмотр / экспорт] ──► Пользователь (подача вне приложения)
```

- **Поток данных:** профиль пользователя + сырые документы → парсинг документов → объединённый контекст → LLM → структурированные поля заявки → валидация → сохранение/экспорт.
- **Компоненты:** клиент (приложение Т-Путешествия), бэкенд визового помощника (API), сервис документов (парсинг/OCR), сервис LLM, хранилище профилей и аудит-логов.

### 4.2 Integration Points

- **Auth:** использование существующей авторизации Т-Путешествия для привязки профиля и синхронизации (TBD: OAuth/токены).
- **Backend Т-Путешествия:** API для сохранения/загрузки визового профиля, метрики использования помощника (события аналитики).
- **Внешние API:** LLM-провайдер, при необходимости — отдельный OCR/document API; все вызовы через бэкенд, ключи не на клиенте.
- **Хранение:** профили и метаданные заявок — БД (TBD: существующая инфраструктура Т-Путешествия или отдельный сервис); файлы документов — защищённое хранилище с ограничением доступа и сроком хранения (см. Security).

### 4.3 Security & Privacy

- **Персональные данные:** визовый профиль и документы относятся к категории особо чувствительных; шифрование при хранении и при передаче (TLS); минимальный объём данных и срок хранения согласно политике и согласию пользователя.
- **Доступ:** только владелец профиля имеет доступ к своим данным; персонал — только при необходимости поддержки, с аудитом доступа.
- **Соответствие:** учёт требований 152-ФЗ (если применимо), GDPR при наличии пользователей из ЕС; явное согласие на обработку ПД и хранение документов.
- **LLM:** не передавать в публичные модели данные, которые не должны покидать периметр; при использовании облачного LLM — соглашения и регионы размещения, соответствующие политике компании.
- **Аудит:** логирование фактов доступа к профилю и экспорта заявок (без сохранения содержимого в логах в долгосрочную перспективу).

## 5. Risks & Roadmap

### 5.1 Phased Rollout

| Фаза | Содержание | Критерий перехода |
|------|------------|-------------------|
| **MVP** | Одна страна (например, Шенген), один тип визы; ручной ввод профиля + загрузка паспорта/фото; заполнение формы через LLM; сохранение профиля локально; проверка обязательных полей | Стабильная точность заполнения ≥ 95%, отсутствие критичных инцидентов безопасности |
| **v1.1** | Все четыре страны (США, Шенген, UK, Япония); несколько типов виз; расширенный парсинг документов; облачное сохранение профиля и синхронизация | Выполнение Success Criteria по повторному использованию и конверсии |
| **v2.0** | Улучшение UX (подсказки по шагам, чек-листы по странам); экспорт в форматы, пригодные для загрузки на официальные порталы (где допустимо); расширенная аналитика и A/B тесты | Рост метрик удовлетворённости и конверсии по плану |

### 5.2 Technical Risks

| Риск | Вероятность/влияние | Митигация |
|------|----------------------|-----------|
| Ошибки LLM в полях (опечатки, неверный формат) | Высокое влияние | Строгая валидация по правилам страны; обязательный предпросмотр и редактирование пользователем; тесты на регрессию |
| Изменение форм/требований консульств | Среднее | Версионирование шаблонов и маппингов; мониторинг официальных источников; процесс обновления без деплоя |
| Задержки/недоступность LLM API | Среднее | Таймауты, повторные попытки, понятные сообщения пользователю; при необходимости — кэш последнего успешного результата |
| Рост затрат на LLM при масштабировании | Среднее | Ограничение размера контекста, кэширование промптов и ответов для типовых сценариев, мониторинг стоимости |
| Утечка или неправомерный доступ к ПД | Критичное | Шифрование, минимальный срок хранения, аудит, соответствие 152-ФЗ/GDPR |
